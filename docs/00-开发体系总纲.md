# 00 · 开发体系总纲

本文是整套开发体系的"地基"，说明 **架构怎么分、人怎么协作、代码怎么从一个想法走到院内服务器**。

---

## 1. 设计目标与约束

| 目标 | 我们的做法 |
|------|-----------|
| 多人并行开发不同场景，互不踩脚 | Monorepo + 场景目录隔离 + 强制 Code Owner |
| 场景之间相互关联（共享登录、患者档案、AI 能力） | 共享库 `packages/` + 统一网关 `services/gateway` |
| 新手也能上手（多数人是 Claude Code 新手） | 每层 `CLAUDE.md` 规范 + 标准 SOP + 模板化新建场景 |
| 私有化部署到院内（数据不出院） | 全容器化 + 离线安装包 + 一键升级脚本 |
| 医疗合规（等保 / 数据安全） | 数据分级、审计日志、最小权限、提交前脱敏检查 |

**技术栈定调**：

- 前端：TypeScript + React + **Next.js**（统一框架，便于共享组件与鉴权）
- 后端：Python + **FastAPI**（统一框架，统一 OpenAPI 契约）
- 包管理：JS 用 **pnpm workspace**，Python 用 **uv**（或 Poetry）
- 任务编排：**Turborepo**（缓存构建/测试/lint）
- 容器：Docker；编排：院内首选 **Docker Compose**（运维简单），规模大时切 K8s
- 代码托管：院内自建 **GitLab**（私有化场景标配）

> 为什么 Monorepo：你的场景"相互关联"。Monorepo 让共享类型、共享组件、跨场景重构在**一次提交**里完成，CI 一次跑通，版本天然对齐。代价是仓库变大、需要约定隔离——这正是本体系要解决的。

## 2. 架构全貌

```
                         院内用户 / 医生 / 患者端
                                  │
                          ┌───────▼────────┐
                          │   统一网关       │  services/gateway
                          │  鉴权·路由·限流  │  (FastAPI / 也可 APISIX)
                          └───────┬────────┘
            ┌──────────────┬──────┴───────┬──────────────┐
       ┌────▼─────┐   ┌────▼─────┐   ┌────▼─────┐   ┌────▼─────┐
       │ 场景 001 │   │ 场景 002 │   │ 场景 003 │   │  ...     │
       │ 前端+后端 │   │ 前端+后端 │   │ 前端+后端 │   │          │
       └────┬─────┘   └────┬─────┘   └────┬─────┘   └────┬─────┘
            └──────────────┴──────┬───────┴──────────────┘
                          ┌───────▼────────┐
                          │   共享能力层      │  packages/
                          │ 类型·UI·SDK·鉴权 │  + 平台服务(patient/ai/file)
                          └───────┬────────┘
                          ┌───────▼────────┐
                          │  基础设施         │  PostgreSQL / Redis / MinIO
                          └────────────────┘
```

**三层职责**：

1. **场景层（apps + services）**：每个编号场景独立的前端页面 + 后端 API。由场景负责人开发。**只能依赖共享层，不能依赖其他场景。**
2. **共享层（packages + 平台服务）**：跨场景复用的东西——类型定义、UI 组件、前端 SDK、鉴权、患者档案服务、AI 能力服务、文件服务。改动影响面大，需架构组评审。
3. **基础设施层（infra）**：数据库、缓存、对象存储、网关、监控。由运维 + 架构组维护。

## 3. 角色与分工

| 角色 | 职责 | 权限 |
|------|------|------|
| **平台架构组** | 维护 `packages/`、`services/gateway`、`infra/`、本体系文档；评审共享层改动 | 全仓 Maintainer |
| **场景负责人** | 负责 1~N 个编号场景的端到端开发与质量；是该场景目录的 Code Owner | 自己场景目录 + MR 权限 |
| **场景开发者** | 在某场景下开发具体功能 | 提 MR，需负责人评审 |
| **运维/部署** | 离线打包、院内安装、升级、监控 | infra + 部署流水线 |

> **关键规则**：`packages/`、`services/gateway`、`infra/` 的改动**必须**经架构组评审（通过 `CODEOWNERS` 强制）。场景内部改动由场景负责人评审即可。

## 4. 一个场景的完整生命周期

```
①领号 → ②建场景骨架 → ③本地开发(Claude Code) → ④自测+门禁
   → ⑤提 MR → ⑥评审 → ⑦合并到 main → ⑧CI 构建镜像
   → ⑨集成测试环境验证 → ⑩打离线包 → ⑪院内部署/升级
```

- ①~⑦ 见 [04 应用场景开发流程](04-应用场景开发流程.md)
- ⑤~⑥ 见 [05 代码评审与质量门禁](05-代码评审与质量门禁.md)
- ⑧~⑪ 见 [06 CI/CD 与私有化部署](06-CICD与私有化部署.md)

## 5. 场景编号与登记规则

- 编号格式：三位数字 `001`、`002` …（预留扩位到四位）。
- 新场景必须先在 [`docs/场景登记表.md`](场景登记表.md) 登记：编号、名称、负责人、依赖的共享能力、状态。
- 一个编号对应一对前后端目录，命名严格固定（见 [02 代码仓库结构](02-代码仓库结构.md)）。
- **不要复用废弃编号**，避免历史镜像/数据混淆。

## 6. 环境分层

| 环境 | 用途 | 数据 | 谁来部署 |
|------|------|------|---------|
| 本地 dev | 个人开发 | mock / 脱敏样例 | 开发者自己 |
| 集成测试 test | 合并后自动部署，联调 | 脱敏数据 | CI 自动 |
| 预发 staging | 上线前验证（镜像院内一致） | 脱敏数据 | 运维 |
| 生产 prod（院内） | 真实使用 | 真实患者数据，不出院 | 运维，离线升级 |

> 私有化关键：**test/staging 用的镜像，必须和最终进院内的镜像完全一致**（同一 tag、同一 digest）。绝不在院内重新构建。
