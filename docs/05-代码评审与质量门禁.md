# 05 · 代码评审与质量门禁

> 两道防线：**自动门禁**（机器把关，CI 红绿）+ **人工评审**（人把关，业务与设计）。
> 两道都过，才能进 main。

---

## 1. 自动门禁（CI 必须全绿）

每个 MR 自动跑，任一不过则禁止合并：

| 门禁 | 工具 | 作用 |
|------|------|------|
| 代码风格 | ESLint + Prettier（前端）/ Ruff（后端） | 统一风格 |
| 类型检查 | tsc（前端）/ mypy（后端） | 防类型错误 |
| 单元测试 | Vitest（前端）/ pytest（后端） | 行为正确 |
| 覆盖率 | 同上 | 新增代码覆盖率 ≥ 70%（场景核心逻辑 ≥ 80%） |
| 依赖边界 | ESLint `no-restricted-imports` + 自定义脚本 | 禁止跨场景直接 import |
| 合规扫描 | 自定义脚本 + gitleaks | 禁止密钥/患者数据进仓库 |
| 构建 | turbo build | 确保能构建出镜像 |
| 安全 | `pnpm audit` / `pip-audit` | 已知漏洞依赖 |

> 只跑受影响的包（Turborepo 按依赖图增量），所以 CI 很快。

### 本地先跑，别让 CI 替你发现

```bash
pnpm run check          # 跑全套门禁（提交前必跑）
pnpm run check --filter=scenario-005-*   # 只跑你的场景，更快
```

## 2. 人工评审

### 谁来评审（CODEOWNERS 自动指派）

| 改动范围 | 必需评审人 |
|----------|-----------|
| 某场景内部 | 该场景负责人 |
| `packages/*`、`services/gateway`、平台服务 | 架构组（至少 1 人） |
| `infra/*`、CI | 运维 + 架构组 |
| 安全/鉴权/患者数据相关 | 架构组 + 安全负责人 |

### 评审检查清单（Reviewer 照此看）

**正确性**
- [ ] 实现是否符合该场景 CLAUDE.md 描述的业务？
- [ ] 边界条件（空、越权、并发、异常）是否处理？
- [ ] 有没有对应测试，且测试真的在测行为？

**架构与协作**
- [ ] 有没有违反依赖边界（跨场景 import）？
- [ ] 该复用的有没有复用（患者/鉴权/文件/AI 走平台服务）？
- [ ] 接口改动是否更新了 `shared-types`、是否向后兼容？

**合规与安全**（医疗项目重点）
- [ ] 敏感字段（身份证、手机号、病历）是否脱敏/未入日志？
- [ ] 是否做了鉴权与最小权限校验？
- [ ] 审计日志是否记录关键操作（谁、何时、对哪个患者做了什么）？

**可维护性**
- [ ] 命名、结构是否符合规范？是否有无谓复杂度？
- [ ] 有没有 AI 生成但没必要的冗余代码 / 注释？

### 评审礼仪

- 区分**必须改**（blocking）和**建议**（nit），明确标注。
- 对事不对人；给出理由和更优写法。
- 作者：被要求改的地方改完回复"已处理 + commit 链接"。

## 3. AI 辅助评审

提交前可让 Claude Code 自查（团队已配 `/code-review` 技能，见 [07](07-ClaudeCode进阶-子代理与技能.md)）：

```
/code-review            # 评审当前分支 diff，找 Bug 与可简化项
/security-review        # 安全角度审查（鉴权、注入、数据泄露）
```

> AI 评审是**补充**不是替代。最终对代码负责的是作者和人工 Reviewer。

---

## 4. 合并规则

- ✅ CI 全绿 + 所需评审人 Approve + 无未解决 blocking 评论。
- ✅ 已 rebase 到最新 main（无冲突）。
- ✅ 用 **Squash Merge**，合并信息符合 [03 规范](03-协作规范-分支与提交.md)。
- 合并后自动部署 test 环境。
