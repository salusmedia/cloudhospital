# 02 · 代码仓库结构

> 记住一句话：**场景目录互相隔离，共享的东西放 `packages/`，公共服务放 `services/`。**

---

## 2. 场景命名规则（强约束）

| 类型 | 路径 | 包名 |
|------|------|------|
| 场景前端 | `apps/scenario-XXX-frontend/` | `@hospital/scenario-XXX-frontend` |
| 场景后端 | `services/scenario-XXX-backend/` | `scenario_XXX_backend`（Python 包名用下划线） |

- `XXX` 是三位编号，与 [场景登记表](场景登记表.md) 一致。
- **不允许**出现 `apps/my-new-feature/` 这种非编号目录（门户 `portal` 等平台级例外，需架构组批）。
- 新建场景**不要手动复制**，用脚手架：`pnpm run new:scenario`（见 [04](04-应用场景开发流程.md)）。

## 3. 依赖规则（最重要）

```
✅ 允许                          ❌ 禁止
场景 → packages/               场景 A → 场景 B（直接 import）
场景 → 平台服务(走 HTTP/网关)    场景 → 另一个场景的内部模块
packages → packages            packages → 场景
平台服务 → py-common           共享层 → 业务场景
```

**场景之间需要协作怎么办？**

1. 需要对方的**数据/能力** → 通过**平台服务**或对方暴露的 **HTTP API**（经网关），不直接 import 代码。
2. 需要对方的**类型/契约** → 放进 `packages/shared-types`，双方都从这里引用。
3. 需要复用对方的**组件/逻辑** → 抽到 `packages/ui` 或 `packages/sdk`，由架构组评审。

> 这条规则用 ESLint（`no-restricted-imports`）+ CODEOWNERS + CI 检查共同强制。违反会在 CI 红灯。

## 4. 共享类型怎么保持前后端一致

后端 FastAPI 自动产出 OpenAPI → 由脚本生成 TS 类型到 `packages/shared-types`：

```bash
pnpm run gen:types       # 读取各后端 OpenAPI，生成/更新共享类型
```

前端通过 `packages/sdk` 调接口，SDK 基于 `shared-types`，**接口一改，类型不匹配会在编译期报错**——这是 Monorepo 的最大红利。

## 5. 每个目录都要有 CLAUDE.md

层级覆盖：根 `CLAUDE.md`（全局规范）→ 场景 `CLAUDE.md`（该场景的领域知识、特殊约束）。Claude Code 会**自动按目录层级合并读取**，越靠近的越具体。

新建场景时脚手架会生成场景 `CLAUDE.md` 模板（见 [`docs/templates/scenario-CLAUDE.md`](templates/scenario-CLAUDE.md)），负责人填写后即生效。
