# 07 · Claude Code 进阶 · 团队配置 / 子代理 / 技能

> 这一篇让团队"用得一致、用得高效"。新手可先跳过，熟练后再回来。

---

## 1. 团队共享配置（已在仓库，对所有人生效）

```
.claude/
├── settings.json          # 共享权限、钩子（提交他人不可关闭的规则）
├── agents/                # 共享子代理
└── skills/                # 共享技能
CLAUDE.md（每层）           # 规范，自动按目录层级合并
```

- **CLAUDE.md** = 给 Claude 的"团队规范 + 领域知识"。根级放全局，场景级放该场景。
- **settings.json** = 权限与自动化。例如：允许常用只读命令免确认、提交前自动跑 lint 钩子。
- 个人偏好放 `settings.local.json`（不提交）。

> 维护这些文件本身也走 MR + 评审，由架构组负责，保证全员一致。

---

## 2. 善用 CLAUDE.md（投入产出比最高）

写好 CLAUDE.md 比反复纠正 AI 高效得多。每个场景的 CLAUDE.md 至少包含：

- 业务是什么、给谁用、核心流程。
- 依赖哪些平台服务、怎么调。
- 领域术语表、数据模型、特殊规则。
- 合规：敏感字段、能不能记日志、鉴权要求。
- 测试与门禁要求。
- "不要做的事"（反例最有用）。

模板见 [`templates/scenario-CLAUDE.md`](templates/scenario-CLAUDE.md)。

---

## 3. 推荐的团队子代理（放 `.claude/agents/`）

子代理是"专门干一类活、带独立上下文"的助手。建议团队统一配：

| 子代理 | 用途 |
|--------|------|
| `scenario-scaffolder` | 按规范生成新场景骨架（前后端 + CLAUDE.md + 示例 + 测试） |
| `api-contract` | 改后端接口后，同步更新 `shared-types` 和 `sdk`，检查破坏性变更 |
| `compliance-reviewer` | 专审医疗合规：脱敏、审计日志、鉴权、最小权限 |
| `test-author` | 为指定模块补齐单测/边界用例 |

> 用法：在对话里说"用 compliance-reviewer 检查这个 MR 的合规问题"。

---

## 4. 推荐的技能 / 命令

仓库已可用的（部分为内置）：

- `/code-review` —— 评审当前 diff，找 Bug 与可简化项（提 MR 前自查）。
- `/security-review` —— 安全审查（鉴权、注入、数据泄露）。
- 团队自建技能放 `.claude/skills/`，比如"按本项目规范新建一个场景""生成离线部署包清单"等。

---

## 5. 协作中的 Claude Code 最佳实践

1. **在项目根启动** `claude`，它才能读到全部 CLAUDE.md 层级。
2. **小步 + review diff + 跑测试**，三件套（见 [01](01-环境搭建与ClaudeCode入门.md)）。
3. **让它引用规范**：明确说"遵循 xxx/CLAUDE.md"。
4. **危险动作人来确认**：push、合并、删分支、删数据、对外发布——不交给 AI 自动做。
5. **不要喂真实患者数据**给 AI；用脱敏样例。
6. 复杂任务可让它**先出计划**（plan），你确认后再执行。

---

## 6. 让 AI 帮你维护这套体系

- 新场景：让 `scenario-scaffolder` 生成，省去手抄、避免命名错误。
- 接口变更：让 `api-contract` 自动同步类型，防前后端漂移。
- 文档更新：规范变了，让 Claude 同步更新相关 CLAUDE.md 和 docs（仍走 MR 评审）。

> 目标：让"规范"既是给人看的，也是给 AI 执行的——人和 AI 用同一套规则，协作才一致。
