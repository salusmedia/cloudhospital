# 04 · 应用场景开发流程（标准 SOP）

> 这是开发一个编号场景，从"领号"到"合并进 main"的标准操作流程。
> 每个开发者每次开发都照这个走。

---

## 步骤总览

```
①领号登记 → ②生成场景骨架 → ③写场景 CLAUDE.md → ④用 Claude Code 开发
   → ⑤本地门禁自测 → ⑥提 MR → ⑦评审与修改 → ⑧合并 → ⑨清理分支
```

---

## ① 领号 & 登记

1. 在 [`docs/场景登记表.md`](场景登记表.md) 申请一个编号（如 `005`）。
2. 填写：编号、名称、负责人、它依赖哪些**平台服务/共享能力**、它会**对外暴露**什么 API、状态=开发中。
3. 与架构组确认它和其他场景的关联点（避免重复造轮子）。

> 关联设计要前置：如果场景 005 需要患者档案，应复用 `platform-patient`，而不是自己存一份患者表。

---

## ② 生成场景骨架（不要手抄）

```bash
git switch main && git pull
pnpm run new:scenario        # 交互式：输入编号 005、名称、是否需要前端/后端
```

脚手架会生成：

```
apps/scenario-005-frontend/        # Next.js 子应用 + CLAUDE.md 模板
services/scenario-005-backend/     # FastAPI 服务 + CLAUDE.md 模板
```

并自动：注册到 workspace、接入网关路由、接入统一鉴权、生成示例接口+示例页面+示例测试。

> 如果暂时还没有脚手架脚本，让 Claude Code 参照已有场景（如 001）和 [模板](templates/scenario-CLAUDE.md) 帮你生成，**但目录命名必须符合规范**。

---

## ③ 写好场景 CLAUDE.md（关键，决定 AI 质量）

打开 `services/scenario-005-backend/CLAUDE.md` 和前端的，按 [模板](templates/scenario-CLAUDE.md) 填：

- 这个场景**是什么、给谁用、核心业务流程**。
- 依赖哪些平台服务、调用方式。
- 领域术语、数据模型、特殊业务规则。
- 合规要求（哪些是敏感字段、能不能记日志）。
- 测试要求。

> CLAUDE.md 写得越清楚，Claude Code 写出的代码越贴合业务、越少返工。**这一步别偷懒。**

---

## ④ 开发（与 Claude Code 协作）

```bash
git switch -c feature/scenario-005-初始功能
claude     # 在项目根启动
```

协作要点（新手详见 [01](01-环境搭建与ClaudeCode入门.md) 第二部分）：

- **小步**：一次一个接口/一个页面；每步看 diff、跑测试。
- **让它引用规范**："遵循 services/scenario-005-backend/CLAUDE.md 实现……"
- **复用优先**：需要登录、患者数据、文件、AI 能力，让它走 `packages/sdk` / 平台服务，**别重造**。
- **同步契约**：后端接口定稿后 `pnpm run gen:types` 更新共享类型，前端立即可用。

---

## ⑤ 本地门禁自测

```bash
pnpm run check --filter=scenario-005-*    # lint + typecheck + 单测（你的场景）
pnpm run e2e --filter=scenario-005-*      # 端到端（若有）
```

全绿才提交。详细门禁项见 [05](05-代码评审与质量门禁.md)。

---

## ⑥ 提交 & 提 MR

```bash
git add -p
git commit -m "feat(scenario-005): 初始随访问诊流程"
git push -u origin feature/scenario-005-初始功能
```

到 GitLab 提 MR（模板自动带出），指派场景负责人；若动了共享层，CODEOWNERS 会自动加架构组。

---

## ⑦ 评审与修改

- 按评审意见在**同一分支**继续改、push，MR 自动更新。
- CI 必须全绿。
- 评审通过 + 门禁通过 → 可合并。

---

## ⑧ 合并 & ⑨ 清理

- 用 **Squash Merge**（把一个 MR 压成一个干净提交进 main），保持主干历史清爽。
- 合并后删除特性分支。
- 合并触发 CI 自动构建镜像并部署到 **test 环境**，去验证。

---

## 场景之间联调怎么做

1. 你的场景调别人的场景/平台服务：在 test 环境联调（各场景都已部署）。
2. 本地联调：`pnpm run dev --filter=scenario-005-* --filter=platform-patient` 同时起依赖项。
3. 契约对不上 → 看 `shared-types` 是否最新，重跑 `gen:types`。

---

## 常见返工点（提前避免）

- ❌ 自己存了一份患者/用户数据 → 应复用平台服务。
- ❌ 直接 import 别的场景代码 → CI 会红，改走共享层或 HTTP。
- ❌ 敏感字段进了日志 → 合规检查会拦，用脱敏工具（`py-common`）。
- ❌ 接口改了没更新 `shared-types` → 前端编译报错。
