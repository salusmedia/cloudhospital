# 03 · 协作规范 · 分支与提交

> 目标：多人并行，**main 永远可部署**，谁改了什么一目了然，冲突最少。

---

## 1. 分支模型（基于主干 + 短生命周期特性分支）

```
main         ← 永远可部署，受保护，只能通过 MR 合入
  ├── feature/scenario-001-add-followup   ← 你的工作分支（短命，几天内合掉）
  ├── feature/scenario-002-fix-pagination
  ├── fix/gateway-timeout
  └── release/v1.4.0                       ← 发版时从 main 切，打 tag
```

- **不要长期分支**：特性分支尽量 1~3 天合掉，越久越难合。
- **每天 `git pull` rebase main**，让分支跟上主干。
- 受保护分支：`main`、`release/*` 不能直接 push，只能 MR。

### 分支命名

```
feature/scenario-XXX-简短描述      # 新功能
fix/scenario-XXX-简短描述          # 修 Bug
fix/<模块>-简短描述                 # 平台/网关等非场景
chore/...  docs/...  refactor/...  # 杂项/文档/重构
```

例：`feature/scenario-001-add-followup-query`、`fix/gateway-timeout`。

---

## 2. 提交信息规范（Conventional Commits）

```
<type>(<scope>): <简短中文描述>

[可选正文：为什么这么改]

[可选脚注：关联 issue / 破坏性变更]
```

- **type**：`feat` 新功能 / `fix` 修复 / `refactor` 重构 / `docs` 文档 / `test` 测试 / `chore` 杂务 / `perf` 性能。
- **scope**：场景编号或模块名，如 `scenario-001`、`gateway`、`shared-types`。

示例：

```
feat(scenario-001): 新增按日期查询随访记录接口
fix(gateway): 修复上游超时未透传 504 的问题
refactor(packages/sdk): 统一错误处理拦截器
```

> 提交粒度：**一个提交一件事**。不要把 5 个不相关改动塞一个 commit。可用 `git add -p` 拆分。

---

## 3. Merge Request 规范

- 标题 = 主提交信息；正文用 [MR 模板](templates/PR模板.md)（仓库已配，自动带出）。
- **小 MR 优先**：一个 MR 最好 < 400 行 diff，便于评审。
- 必须：关联场景编号、说明影响面、附自测结论、CI 全绿。
- 指派：场景内改动 → 场景负责人；共享层改动 → 架构组（CODEOWNERS 自动要求）。

详见 [05 代码评审与质量门禁](05-代码评审与质量门禁.md)。

---

## 4. 冲突处理（Monorepo 高频）

因为大家在一个仓库，最常见冲突是 **lockfile** 和 **shared-types**：

- `pnpm-lock.yaml` / `uv.lock` 冲突：**不要手动改**，删掉冲突后重跑 `pnpm install` / `uv lock` 重新生成。
- `packages/shared-types` 冲突：通常是接口同时变更，**两位作者沟通**后重跑 `pnpm run gen:types`。
- 日常 `git pull --rebase` 比 merge 更干净，减少无意义的 merge commit。

---

## 5. 大家同时改"共享层"怎么办

共享层（`packages/`、`services/gateway`、平台服务）是协作热点：

1. 改共享层前，**先在群里/issue 同步**，避免两人同时大改。
2. 共享层改动**向后兼容优先**：加字段而非改字段，废弃用 `@deprecated` 标注过渡。
3. 破坏性改动必须：架构组评审 + 通知所有依赖方 + 同一 MR 内修复所有调用点（Monorepo 的好处就是能一次改全）。

---

## 6. 节奏建议

- 每日：早上 `pull --rebase`，下班前把当天工作提交并 push（哪怕是 WIP 分支）。
- 每个 MR 合并后立刻删分支（GitLab 可设置自动删）。
- 别攒大改动；攒得越久，越难评审、越容易冲突。
